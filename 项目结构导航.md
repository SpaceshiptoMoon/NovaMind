## 后端架构

> 所有功能模块位于 `backend/src/` 下，每个模块遵循 DDD 分层结构：
> `api/ (路由+DI+异常) → services/ (业务逻辑) → repository/ (数据访问) → models/ (ORM) → schemas/ (Pydantic)`

### 核心基础设施 (core)

| 功能 | 文件路径 | 说明 |
|------|---------|------|
| 应用工厂 | `backend/src/core/middleware/app_factory.py` | `create_app()` 组装中间件、异常处理、路由、限流 |
| 路由注册 | `backend/src/core/middleware/router_manager.py` | 所有 API Router 懒加载注册到 `/api/v1` |
| 启动管理 | `backend/src/core/middleware/startup_manager.py` | 生命周期管理：Redis → 建表 → 模块初始化 → arq Worker |
| 异常基类 | `backend/src/core/middleware/base_exception_handler.py` | `BaseAPIError` + 自动 HTTP 状态码映射 |
| 全局异常 | `backend/src/core/middleware/exceptions.py` | 注册所有模块的异常处理器 |
| 链路追踪 | `backend/src/core/middleware/trace_middleware.py` | ASGI 中间件，每请求生成 trace_id |
| 结构化日志 | `backend/src/core/middleware/structured_logging.py` | structlog JSON 日志 + 文件轮转 |
| 健康检查 | `backend/src/core/middleware/health_check.py` | `/health` 系列端点 (DB/Redis/ES/MinIO) |
| 限流 | `backend/src/core/middleware/rate_limit.py` | slowapi 限流 (登录 5/min、注册 3/min 等) |
| 数据库基类 | `backend/src/core/database/base.py` | `BaseModel` (id/created_at/updated_at 自动) |
| 数据库连接 | `backend/src/core/database/database.py` | `get_db()` 异步会话管理 |
| 密码哈希 | `backend/src/core/auth/hashing.py` | Argon2 哈希 + 异步包装 |
| 安全校验 | `backend/src/core/security/config_validator.py` | 生产环境启动安全检查 (JWT/密码/CORS) |

**健康检查端点:**
| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/health` | 基础存活检查 |
| GET | `/health/detailed` | 详细检查 (DB/Redis/ES/MinIO) |
| GET | `/health/ready` | 就绪检查 (仅 DB) |
| GET | `/health/live` | 存活检查 (无依赖) |

## 关键数据流

### 文档处理管道
```
用户上传 → DocumentService.upload_document (MinIO 存储)
         → arq Worker.enqueue
         → DocumentProcessor (解析+切片)
         → EmbeddingService (Redis 缓存向量化)
         → QuestionGenerationService (可选)
         → ElasticsearchClient (批量索引)
```

### AI 对话流 (SSE)
```
前端 chatStream() → createSSEStream (Fetch API)
  → AIChatService.chat_stream()
    → 准备上下文 (附件注入+系统提示)
    → BaseLLM.generate_text_stream()
    → 心跳保活 + 流式输出
    → 保存 AI 回复
```

### Agent ReAct 循环
```
AgentEngine.run() (最多 max_iterations 轮)
  → LLM.generate_with_tools()
  → 若返回 tool_calls → ToolExecutor 路由执行
    → 内置工具 (KnowledgeSearch/WebSearch/CodeExecution)
    → 或 MCP 外部工具
  → 追加工具结果到消息
  → 继续循环直到纯文本输出
```

### 搜索流程
```
SearchService.search()
  → 查询改写 (HyDE/子查询)
  → EmbeddingService 生成查询向量
  → ElasticsearchClient 搜索 (BM25/向量/混合 + RRF)
  → 分数归一化 + 阈值过滤
  → Rerank (可选)
  → LLM 回答生成 (可选)
  → Redis 缓存结果
```

### 简历挖掘管道 (V2 三阶段)
```
用户上传简历 → POST /apps/resume/upload (异步后台执行)

S1-S4: 简历解析 (ResumeParser)
  文本提取 → 段落分割 → 6路并行LLM解析 → 交叉验证组装 StructuredResume

S4.5: 简历概述 → S5: JD分析 (可选)

S6: 工作-项目合并 (ResumeAnalyzer)
  project_experience + work_experience → WorkProjectUnit[] (按公司名匹配)

S6.5: 公司/岗位背景补充
  搜索引擎查询 "{公司名} {岗位} 技术方向"
  → LLM 综合生成 company_industry / position_context / industry_context
  → 失败时降级到纯 LLM 推断

S7: 复杂度评估 + 追问策略
  LLM 评估每个 WorkProjectUnit 4维度 (技术栈/角色/描述/权重)
  → 自动分配 2-8 轮追问
  → 以工作单元为核心组织知识点和追问链

S8: 技术前置学习 (Part 1)
  简历技术栈 → 学习 Q&A (基础→场景→进阶→对比) + 踩坑点

S9: 中间报告 → MinIO 存储

S10: 项目深度追问 (Part 2 核心, AutoProbingEngine)
  并行追问所有 KnowledgePoint (信号量限制并发数)
  Round 1: 做法 (你怎么做的？)
  Round 2: 原因 (为什么这么做？)
  Round 3: 结果 (得出什么结果？)
  Round 4: 困难 (遇到什么困难？)
  Round 5+: 基于上一轮回答自由追问
  每轮必须引用上一轮回答，带重试降级 (主模型3次 → 用户备选模型)

S11: 面试准备建议 + 简历优化建议 (Part 3)
  低分回答标记为薄弱点 → 薄弱点分析 + 高概率追问点 + STAR法改写建议

S12: 最终三段式报告组装 → 覆盖 MinIO → 更新会话状态为 COMPLETED
```
