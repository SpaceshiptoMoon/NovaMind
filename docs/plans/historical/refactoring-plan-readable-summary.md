# 文档处理管道重构计划摘要

> 历史说明：本摘要整理自 2026-07-06 的早期计划文档 [`refactoring-plan.md`](./refactoring-plan.md)。
> 原始文档现已整理为可读版本，但如果只是理解历史方案，请优先阅读本摘要。

## 文档定位

这份历史计划关注的是知识库文档处理链路的早期重构方向，核心目标包括：

- 把文档元数据、处理任务状态、知识库配置三类职责拆开
- 统一不同模态文件进入切分与索引前的处理链路
- 让处理配置与处理结果具备可追溯性
- 逐步把临时态的 Redis 任务追踪迁移到可持久化的任务模型上

## 当时识别出的主要问题

- `Document` 同时承担文件元数据和处理状态，职责混杂。
- 文本、音频、视频三类文件的“解析后再切分”路径不统一。
- 某些切分配置放在 `parsing.audio` 等位置，语义不清晰。
- 任务状态主要依赖 Redis，缺少持久化任务记录。
- 后端仓库与前端页面中存在一批已经接近无引用的旧逻辑和死代码。

## 计划中的目标结构

这份历史方案想把流程拆成三层：

1. `Document`
   只保留文件元数据、存储位置、软删除信息等。
2. `DocumentTask`
   单独记录每次处理任务的状态、时间线、错误信息、配置快照和处理结果。
3. `KnowledgeBase.config`
   作为处理管道的配置来源，承载解析、切分、问题生成等结构化参数。

## 历史方案中的关键设计点

### 1. 解析与切分之间引入统一桥梁

历史方案提出：无论原始文件是文本、图片、音频还是视频，解析阶段都尽量先产出一份统一的 Markdown 全文，再由切分器继续处理。

设计动机包括：

- 便于审计进入切分器的实际文本内容
- 便于复用同一份解析结果重新切分
- 新增文件类型时，只需新增“原始文件到 Markdown”的解析器

### 2. 引入持久化任务模型

历史方案建议为每次处理建立 `DocumentTask`，记录：

- `status`
- `job_id`
- `pipeline_config`
- `step_progress`
- `pipeline_result`
- `error_message`
- `retry_count`
- `queued_at` / `started_at` / `completed_at`

它反映的是当时对“文档处理应该是可追踪任务，而不是若干零散字段”的设计方向。

### 3. 配置分层要更清晰

历史文档强调过：

- 解析相关参数应放在 `parsing`
- 切分相关参数应放在 `splitting`
- 音频/视频等模态专属参数应有更清楚的嵌套边界

这部分虽然不是当前代码的最终结构定义，但能说明后来为什么持续在知识库配置文档中收敛字段边界。

## 文档中列出的实施范围

原计划覆盖了：

- 后端模型与仓储层调整
- 文档处理与 worker 流程重构
- API 返回结构调整
- 前端 `Document` / `Task` 相关页面与状态展示改造
- Redis `task_tracker` 清理
- 最终验证与风险审查

## 历史风险与未决点

原文还列出了几类重点风险：

- 大量逻辑直接依赖 `Document.status`
- 前端页面对处理状态字段的耦合
- worker 中可能存在直接依赖旧表或旧字段的原始 SQL
- 统计查询对旧状态字段的依赖

同时保留了一些未定设计，例如：

- `SplittingConfig` 的最终结构
- Elasticsearch metadata 的保留方式
- 取消任务的实现策略
- 并发防护和软删级联策略
